# CONTEXT.md — fried-plantains domain glossary

## Alert

A record created by the detection runner when a detection rule matches events in the data lake. An alert is composed of two parts:

- **Alert event** (immutable): written to `DeviceAlertEvents` Parquet partition. Represents the detection fact — what fired, when, with what severity. Queryable via KQL alongside all other MDE tables.
- **Triage state** (mutable): stored in a separate SQLite file (`storage/alerts/triage.db`). Tracks analyst workflow: `status` (open → investigating → closed) and `notes`. Joined into the alert API response at read time.

## Detection Rule

A YAML file in `detections/rules/` that defines a KQL/SPL/SQL query and metadata (severity, MITRE tags, `mde_portable` flag). Rules are data, not code. The detection runner executes rules on a schedule and creates Alerts when they match.

## MDE Portability

A rule marked `mde_portable: true` must use only table names and column names defined in `backend/schema/mde_tables.py`, with exact MDE casing. fried-plantains-generated alerts are written to `DeviceAlertEvents` in the MDE schema so alert-hunting queries are also portable.

## Alert Store

A shared service module (`backend/services/alert_store.py`) that owns all alert I/O — writing to `DeviceAlertEvents` Parquet and reading/writing triage state in SQLite. Both `api/alerts.py` and `detection_runner.py` import from it. Neither imports the other. The detection runner loop itself remains monolithic (scheduling + execution + persistence in one task) — splitting it further is premature at homelab scale.

## DeviceAlertEvents

The MDE table that stores alert records. In fried-plantains, this table receives both:
- Alerts ingested from a real MDE tenant (external source)
- Alerts generated by the fried-plantains detection runner (`ServiceSource = "fried-plantains"`)
